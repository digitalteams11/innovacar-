# SEO route inventory

Audited directly from `frontend-web/src/App.tsx` (the actual `<Routes>` table) on 2026-07-16 — not from assumed marketing-site routes. **This app has no marketing website today.** `innovacar.app` serves the SaaS product itself: `/` is the authenticated Dashboard (`ProtectedRoute` → redirects to `/login` when unauthenticated). There is no landing page, `/features`, `/pricing`, `/about`, `/security`, `/privacy`, `/terms`, or `/faq` page anywhere in this codebase or in the Electron `frontend_desktop` app.

Routing is `HashRouter` end-to-end (`frontend-web/src/main.tsx`). Every real URL path returns the identical `index.html` shell — react-router only reads the `#/...` fragment, which the server/CDN never sees. This has two consequences used throughout this inventory:
- A path like `https://innovacar.app/login` does **not** render the login page today — it renders whatever `/` (real pathname) resolves to, i.e. the dashboard-or-redirect-to-login shell. Only `https://innovacar.app/#/login` renders it.
- Vercel path-matched rules (headers, robots-style disallow) cannot distinguish one hash route from another, because the server only ever sees the pre-`#` path.

## What was actually implemented against this inventory

- `frontend-web/src/components/seo/SeoHead.tsx` + `robotsPresets.ts` — reusable per-page meta component, applied to every route above that has a real page component (`Login`, `Register`, `ForgotPassword`, `VerifyResetCode`, `ResetPassword`, `VerifyEmail`, `PublicContact`, `PublicContract`, `InspectionCapture`).
- `frontend-web/index.html` ships `<meta name="robots" content="noindex,follow,noarchive">` in the raw HTML shell as the safe default — the dashboard/private majority of the app is protected even before `SeoHead` runs, and even on the very first paint search engines see.
- `frontend-web/public/robots.txt` — see file; deliberately does **not** attempt path-based `Disallow` rules (see HashRouter note above for why they'd be no-ops), relies on per-page `noindex` instead, per Google's own guidance.
- `frontend-web/public/sitemap.xml` — generated by `scripts/generate-sitemap.mjs` from `src/seo/public-routes.json`. **Currently empty** (valid `<urlset>` with 0 entries) because the one indexable page (`/contact`) only exists at a hash URL, which sitemaps cannot contain.
- `vercel.json` — site-wide `X-Robots-Tag: noindex, follow, noarchive` header (excluding `/sitemap.xml` and `/robots.txt`), reinforcing the HTML meta default at the HTTP layer.

## Part 3 — public marketing site (added 2026-07-21)

Three real, hash-free, indexable pages now exist, served without touching HashRouter or the private app at all:

- `/` — homepage (`src/marketing/pages.tsx` → `HomePage`)
- `/fonctionnalites` — features page
- `/tarifs` — pricing page (qualitative tiers + one real anchor price, no invented figures — see file comments for why)

**How it works**: `src/main.tsx` decides at boot, before mounting anything, whether to render the marketing site or the existing `<HashRouter><App/></HashRouter>` tree. Any URL carrying a `#/...` fragment always goes to the private app, unconditionally — every existing deep link, email link, and bookmark is unaffected. Only a hash-free navigation to one of the three paths above, with no auth token in `localStorage`, renders the public marketing site instead; a logged-in user who bookmarked `/` still lands in the app.

**Prerendering**: `scripts/prerender-marketing.mjs` (run after `vite build`, before `verify:seo`) compiles `src/marketing/pages.tsx` with `vite`'s `transformWithOxc` and renders each page to static HTML via `react-dom/server`, injecting it into a copy of the already-built `dist/index.html` (so hashed asset links stay correct) along with a page-specific `<title>`, description, canonical, `index,follow` robots, Open Graph tags, and (homepage only) a `SoftwareApplication` JSON-LD block. This is the spec's "Acceptable" prerender option — no new SSR/SSG framework was introduced. `scripts/verify-seo-dist.mjs` now asserts all three pages exist with exactly one robots/canonical tag each and non-trivial static content.

`vercel.json` excludes `/fonctionnalites` and `/tarifs` from the SPA catch-all rewrite (so Vercel serves the prerendered static files, not the generic shell) and excludes `/`, `/fonctionnalites`, `/tarifs` from the site-wide `X-Robots-Tag: noindex` header.

## Part 4 — fixed the `/sitemap` noindex bug (2026-07-24)

Google Search Console reported `https://innovacar.app/sitemap` (no `.xml`) as
"Excluded by noindex tag." Root cause: that path was never a real route —
it's not in `App.tsx`, not in `src/marketing/pages.tsx`, and wasn't in
`vercel.json`'s rewrite-exclusion list or its `X-Robots-Tag` header
exclusion list. So it fell through to the generic SPA rewrite (served the
`index.html` shell, whose raw HTML default is `noindex,follow,noarchive`)
*and* got the site-wide `X-Robots-Tag: noindex` header — doubly blocked.
`sitemap.xml` itself was never affected; the bug was specifically the
extensionless path.

Fix (`vercel.json` only, no app code changed):
- Added a permanent redirect `/sitemap` → `/sitemap.xml`, so Search Console
  re-crawling that exact URL now gets a 301 straight to the real sitemap.
- Added a permanent `www.innovacar.app` → `https://innovacar.app` redirect —
  the canonical hostname decision (bare apex) is now enforced at the routing
  layer, not just documented.
- Added a host-scoped `X-Robots-Tag: noindex, nofollow, noarchive` header
  rule (`missing: host=innovacar.app`) so any non-production host — a Vercel
  preview/branch deployment — is forced noindex on every path, including the
  marketing pages that are otherwise exempted by path on production.
- `scripts/verify-seo-dist.mjs` now asserts all three rules exist in
  `vercel.json` (by parsing it directly, not just scanning `dist/`) and
  actually runs the path-based noindex rule's regex against `/sitemap.xml`,
  `/robots.txt`, and an arbitrary private path to prove the exclusion works,
  rather than eyeballing the pattern. Also added: duplicate-`<loc>` and
  token/query-string checks on `sitemap.xml` entries.

## Follow-up work not done here (explicitly out of scope this pass)

1. **Multilingual `/en/ /fr/ /ar/` URLs and hreflang.** The 3 pages above are French-only. Do not add hreflang tags until real English/Arabic page content exists (reviewed, not machine-translated at request time) — a self-referencing hreflang with only one language is misleading.
2. **City landing pages and blog** — explicitly deferred; the spec itself warns against thin, template-swapped city pages, so these need real per-city content, not a generator.
3. **Real 1200×630 OG image, favicon PNG set (16x16/32x32/apple-touch/android-chrome)** — still no image-generation tooling available; the marketing pages currently reuse `brand/innovacar-logo.png` as a stopgap `og:image`, which is a logo, not a proper social preview card.
4. **Branded 404 page** — the current `*` route is a client-side redirect to `/` or `/super-admin` by design (existing product behavior); replacing it with a real 404 page is a UX decision, not made unilaterally here.
5. **Legal pages** (privacy/terms/security) — not built; no legal copy was supplied to draft from, and inventing it would be worse than not having the pages.
